## Shared Between Both Lanes (Parent)

## Purpose

Project Portus aims to make Wicket site configuration portable, reviewable, and repeatable.

For this stack, that means one exportable manifest that can:

1. Read configuration from the Wicket-owned stack.
2. Serialize it into a stable file format.
3. Import it safely into another environment.
4. Support history and review (in the future).

This document is a first-pass plan based on the current codebase, not a product spec.

## Executive Summary

### Required Team Decision: Choose One Execution Lane

This plan intentionally offers two execution lanes, and the team must choose one lane together before implementation starts.

- Lane 1: baseline route (Portus-first core implementation).
- Lane 2: accelerated route (Portus orchestration over third-party transfer primitives).
- Decision rule: lane selection is a team decision, not an individual decision, and the team should execute one active lane consistently for the week.
- No lane mixing during hack week: choose one lane and stay on it for the full week, because the team does not have capacity to run both paths.

The stack does not have one configuration system.

It has several:

- `wicket_settings` option array in the base plugin.
- Carbon Fields and some ACF-backed settings in Account Centre.
- standalone plugin options in Gravity Forms and Memberships.
- ACF JSON in the main theme and Account Centre plugin.
- custom post types plus post meta in Memberships.
- WooCommerce product and order meta in Financial Fields and Guest Checkout.
- code-driven config in `wicket-lib-org-roster`.

Because of that, Portus should not start as a full WordPress backup tool.

But based on the intended use, it also cannot stop at raw settings.

Hack week should deliver an MVP that handles Wicket-managed configuration plus content-adjacent artifacts that define how Wicket sites work:

- Wicket base settings.
- Account Centre options.
- Wicket Gravity Forms plugin options.
- Wicket Memberships plugin options.
- Wicket Financial Fields settings.
- selected theme ACF option-page values.
- plugin/theme ACF JSON assets already stored in Git.
- Wicket-managed pages and page references.
- plugin inventory relevant to Wicket delivery.
- selected Wicket-owned CPT/config records that behave like configuration.

Hack-week rule:

- prefer broad capture where the real dependency surface is unknown.
- keep the module system narrow and extensible so new capture surfaces can be added later without redesigning Portus.

It should explicitly defer volatile or content-heavy data:

- WooCommerce orders.
- transient caches.
- bulk-upload jobs.
- user meta session state.
- one-off runtime data.

Working rule:

- include records that define the site Wicket builds.
- exclude records generated by users, purchases, sessions, or operations after launch.

Execution-route note:

- this plan contains two implementation routes:
  - baseline route (Portus-first core implementation)
  - accelerated route (Portus orchestration over third-party transfer primitives)
- the team should choose one route as the active path together, then execute that path consistently for the week.
- mixed execution is not allowed during hack week; one route only.

## What The Wicket Stack Actually Uses

### 1. Base plugin: central settings hub

Primary finding:

- The base plugin reads most shared settings from the single WordPress option `wicket_settings`.

Evidence:

- `wicket_get_option()` and `get_wicket_settings()` read from `get_option('wicket_settings', [])`.
- The admin settings page is built in the base plugin and is extended by other plugins through filters.

Implication:

- `wicket_settings` is the closest thing this stack has to a shared configuration nucleus.
- Portus should treat this option as a first-class export/import surface.

### 2. Account Centre: mixed Carbon Fields + shared base settings + ACF JSON

Primary findings:

- Account Centre uses Carbon Fields `theme_options` for ACC-specific options.
- It also writes environment settings into the same shared `wicket_settings` option when the base plugin is not present.
- It stores block field groups in `includes/acf-json`.

Implication:

- Portus must capture ACC config from two places:
  - `wicket_settings` for shared/base-aligned environment settings.
  - Carbon Fields theme options for ACC-specific settings.
- Portus should not scrape ACC admin screens. It should read the underlying option stores directly.

### 3. Gravity Forms plugin: standalone plugin options, plus native GF data

Primary findings:

- The Wicket GF plugin stores plugin-level settings in options like:
  - `wicket_gf_slug_mapping`
  - `wicket_gf_pagination_sidebar_layout`
  - `wicket_gf_member_fields`
- Form definitions, feeds, confirmations, notifications, and native GF settings live in Gravity Forms tables/posts, not in the Wicket plugin.

Implication:

- There are two separate scopes:
  - Wicket GF plugin config.
  - actual Gravity Forms site configuration.
- For hack week, only the Wicket GF plugin option surface is in scope.
- Native Gravity Forms forms, feeds, confirmations, notifications, and plugin settings are out of scope for the MVP.
- Entry data remains out of scope.

### 4. Memberships plugin: option + CPT/meta-driven config

Primary findings:

- The plugin stores admin settings in `wicket_membership_plugin_options`.
- It also relies heavily on custom post types and post meta for membership configs, tiers, and membership records.

Implication:

- Memberships has both:
  - site-level plugin configuration.
  - business/content configuration represented as CPTs.
- Membership config/tier records are part of Wicket’s implementation work and should be treated as Portus-managed artifacts.
- Membership records created by real customer lifecycle events should remain out of scope.

### 5. Financial Fields: shared settings + Woo product/order meta

Primary findings:

- Plugin-level settings are stored in `wicket_settings`.
- Product-level finance mappings are stored as WooCommerce product meta.
- Dynamic runtime values are written to order item/order meta.

Implication:

- Portus MVP should include plugin-level finance settings from `wicket_settings`.
- Product finance mapping should be considered in scope when the products are part of the Wicket-delivered site configuration.
- For hack week, WooCommerce should start with general WooCommerce settings plus membership-related WooCommerce configuration.
- Order-level finance data is runtime data and should not be imported as environment config.

### 6. Guest Checkout: mixed configuration and runtime state

Primary findings:

- The plugin uses both shared `wicket_settings` keys and legacy standalone options.
- It also writes order meta, user meta, and transient-driven session state.

Implication:

- Only import durable configuration:
  - enabled flags
  - templates
  - token-expiry settings
- Never import user/session/order transient state.

### 7. Main theme: ACF option pages, ACF JSON, block JSON, menus, and theme behavior

Primary findings:

- The non-V1 theme is `wicket-wp-theme`.
- It registers several ACF options pages:
  - Global
  - Header
  - Footer
  - Global Search
  - Theme Styling
- It stores ACF JSON under `acf-json/`.
- It loads ACF block field groups from block directories.

Implication:

- Theme config spans both:
  - Git-tracked schema/assets (`acf-json`, `block.json`, block code).
  - database-backed option-page values.
- Portus MVP should import/export theme ACF option values and related page/menu references used by the implementation.
- Custom pages created as part of the build should be considered in scope where they act as configuration-bearing site assets.
- Git should remain the source of truth for theme code and ACF JSON assets already versioned in the repo.

### 8. Org roster library: mostly code/config filter driven

Primary findings:

- The org roster library uses a code-defined config array in `OrgManConfig`.
- Runtime behavior is expected to be overridden via filters, not mostly through admin settings.
- It has some option/user-meta usage for bulk upload jobs and additional seats state, but that is operational state, not primary site configuration.

Implication:

- Portus should not try to serialize the org roster library’s hard-coded config as if it were database config.
- If Wicket wants org-roster config portable, the better path is to define a formal external config contract and have Portus consume that later.

Proposed top-level structure:

```json
{
  "schema_version": 1,
  "generated_at": "2026-03-27T00:00:00Z",
  "site": {
    "url": "https://example.test",
    "environment": "stage"
  },
  "modules": {
    "site_inventory": {
      "plugins": [],
      "theme": {},
      "dependencies": []
    },
    "wicket_settings": {},
    "account_centre": {
      "carbon_fields": {},
      "acf_options": {},
      "pages": []
    },
    "gravity_forms_wicket_plugin": {},
    "memberships": {
      "plugin_options": {},
      "config_posts": [],
      "tier_posts": []
    },
    "financial_fields": {},
    "guest_checkout": {},
    "theme": {
      "acf_options": {},
      "acf_json_inventory": [],
      "pages": [],
      "menus": []
    },
    "woocommerce": {
      "settings": {},
      "membership_related": {},
      "products": []
    }
  }
}
```

Notes:

- The manifest should separate schema/config from operational state.
- The manifest should separate:
  - environment inventory
  - settings/options
  - config-bearing content artifacts
  - runtime/operational data
- The manifest should remain module-extensible so additional page types, WooCommerce surfaces, CPTs, or plugin adapters can be added later without changing the core contract.
- Each module should have its own extractor and importer.
- Secrets are included by design for this workflow.
- The export must include a prominent sensitive-data warning.

## Risks

### Risk 1: scope explosion

“Configuration” becomes “entire site state” very quickly in WordPress.

Mitigation:

- lock the MVP to durable settings first.

### Risk 2: secrets in Git

`wicket_settings` contains environment credentials and endpoints.

Mitigation:

- export secrets verbatim for the intended workflow.
- add explicit warnings in docs and the plugin UI output.
- assume the manifest must be handled as sensitive material.

### Risk 3: content vs configuration ambiguity

Membership tiers/config CPTs and Woo products are partly “content” and partly “configuration”.

Mitigation:

- treat membership config/tier CPTs as in scope.
- keep WooCommerce limited to settings and membership-related configuration for hack week.

### Risk 4: import side effects

Some plugins may trigger saves, cache flushes, rewrites, or downstream API behavior on import.

Mitigation:

- import through stable APIs where possible.
- add dry-run.
- add per-module transactional logging.
- isolate plugin activation changes to a dedicated import phase with reporting.

### Risk 5: reference mismatch across environments

Pages, CPTs, and linked records may not share numeric IDs between environments.

Mitigation:

- use slugs as canonical import keys
- treat IDs as reference-only data in the manifest
- downgrade any surface without stable slug-like identity to export-only for MVP

### Risk 6: over-trusting stakeholder feedback without decision ownership

Support-team feedback is valuable, but it can also expand scope late.

Mitigation:

- implementation and de-scope decisions stay with:
  - Esteban for core/integration/import safety
  - Alex for domain/import practicality
  - Niruja for product-priority escalation
- Justin flags implementation/delivery gaps
- Pat flags business-facing clarity issues
- final cut calls happen by midday day 2 and midday day 3

## Lane 1: Standalone (Baseline) Path (Parent)

## Proposed Technical Approach

### Core design

- Build Portus as a standalone Wicket plugin for hack week.
- Expose a module registry:
  - extractor interface
  - validator interface
  - importer interface
- Each Wicket-owned package contributes adapters through filters or direct registration.
- Each adapter must be independently discoverable and optional so new surfaces can be added incrementally after hack week.

### Why standalone first

- It reduces hack-week risk against the base plugin.
- It gives more freedom to move fast, reshape internals, and break compatibility while the tool is immature.
- It keeps the experiment isolated until the manifest format and import behavior are proven.
- It can still read and orchestrate base-plugin-owned settings without living inside the base plugin.

### Long-term destination

- The long-term home can still be the base plugin.
- To support that, Portus should be written as if its core will be transplanted later:
  - no tight coupling to plugin file paths
  - no direct assumptions about its final plugin slug
  - service classes separated from bootstrap code
  - import/export logic isolated from admin/CLI wiring
  - adapters registered through interfaces/hooks, not hardcoded bootstrap chains

### MVP interfaces

- `ConfigModuleInterface::key(): string`
- `ConfigModuleInterface::export(): array`
- `ConfigModuleInterface::validate(array $payload): array`
- `ConfigModuleInterface::import(array $payload, array $options = []): ImportResult`

### MVP entry points

#### Example A: Register Portus transfer modules with HyperFields Transfer Manager

```php
use HyperFields\ContentExportImport;
use HyperFields\ExportImport;
use HyperFields\Transfer\Manager;

$manager = new Manager();

// Settings module
$manager->registerModule(
    'settings_core',
    exporter: static function (array $context): array {
        $option_names = $context['option_names'] ?? [];
        $prefix = $context['prefix'] ?? '';

        return [
            'json' => ExportImport::exportOptions($option_names, $prefix),
        ];
    },
    importer: static function (array $payload, array $context): array {
        return ExportImport::importOptions(
            (string) ($payload['json'] ?? ''),
            $context['allowed_options'] ?? [],
            $context['prefix'] ?? '',
            ['mode' => $context['mode'] ?? 'merge']
        );
    },
    differ: static function (array $payload, array $context): array {
        return ExportImport::diffOptions(
            (string) ($payload['json'] ?? ''),
            $context['allowed_options'] ?? [],
            $context['prefix'] ?? '',
            ['mode' => $context['mode'] ?? 'merge']
        );
    }
);

// Structural pages / config CPT module
$manager->registerModule(
    'content_structural',
    exporter: static function (array $context): array {
        $post_types = $context['post_types'] ?? ['page'];
        $options = $context['content_export_options'] ?? [];

        return [
            'json' => ContentExportImport::exportPosts($post_types, $options),
        ];
    },
    importer: static function (array $payload, array $context): array {
        return ContentExportImport::importPosts(
            (string) ($payload['json'] ?? ''),
            $context['content_import_options'] ?? []
        );
    },
    differ: static function (array $payload, array $context): array {
        return ContentExportImport::diffPosts(
            (string) ($payload['json'] ?? ''),
            $context['content_import_options'] ?? []
        );
    }
);
```

Why this helps:

- Portus keeps module ownership and manifest control.
- HyperFields handles low-level transfer/diff/import behavior.

#### Example B: Export settings from common Wicket stack surfaces

```php
$context = [
    'option_names' => [
        'wicket_settings',
        'wicket_membership_plugin_options',
        'wicket_gf_slug_mapping',
        'wicket_gf_pagination_sidebar_layout',
        'wicket_gf_member_fields',
    ],
    'allowed_options' => [
        'wicket_settings',
        'wicket_membership_plugin_options',
        'wicket_gf_slug_mapping',
        'wicket_gf_pagination_sidebar_layout',
        'wicket_gf_member_fields',
    ],
    'prefix' => '', // set only if a module intentionally owns prefixed array keys
    'mode' => 'merge',
];

$settings_export = $manager->export(['settings_core'], $context);
```

Notes:

- for large shared arrays like `wicket_settings`, `merge` mode is safer in hack week.
- use `replace` mode only when the option ownership boundary is fully controlled.

#### Example D: Export and dry-run structural pages + config CPTs

```php
$content_context = [
    'post_types' => [
        'page',        // structural pages
        'my-account',  // account-centre pages where relevant
        // add config CPTs later, for example:
        // 'membership_config',
        // 'membership_tier',
    ],
    'content_export_options' => [
        'post_status' => ['publish', 'draft', 'private'],
        'include_meta' => true,
        'include_private_meta' => false,
        'exclude_meta_keys' => ['_edit_lock', '_edit_last', '_wp_old_slug'],
        'include_content' => true,
        'include_excerpt' => true,
        'include_parent' => true,
    ],
    'content_import_options' => [
        'allowed_post_types' => ['page', 'my-account'],
        'dry_run' => true,
        'create_missing' => true,
        'update_existing' => true,
        'include_meta' => true,
        'meta_mode' => 'merge',
        'include_private_meta' => false,
        'exclude_meta_keys' => ['_edit_lock', '_edit_last', '_wp_old_slug'],
    ],
];

$content_bundle = $manager->export(['content_structural'], $content_context);
$content_diff = $manager->diff($content_bundle, $content_context);
```

Matching behavior from HyperFields:

- content records are matched by `post_type + slug`.
- this aligns with Portus import rule to use slug-first matching, not numeric IDs.

#### Example E: Combined export/validate/import flow in Portus

```php
// 1) Export selected modules
$bundle = $manager->export(['settings_core', 'content_structural'], [
    ...$context,
    ...$content_context,
]);

// 2) Validate (dry-run diff)
$diff = $manager->diff($bundle, [
    ...$context,
    ...$content_context,
]);

// 3) Import only after operator confirmation
$apply_context = [
    ...$context,
    ...$content_context,
];
$apply_context['content_import_options']['dry_run'] = false;

$import_result = $manager->import($bundle, $apply_context);
```

How Portus should present this:

- export result: show module export success + warnings.
- validate result: show module-level dry-run changes and skipped records.
- import result: show created/updated/unchanged/skipped counts per module.

#### Example F: Keep Portus manifest contract stable

Use HyperFields payloads as internal transport while preserving Portus manifest shape:

```php
$portus_manifest = [
    'schema_version' => 1,
    'generated_at' => gmdate('c'),
    'modules' => [
        'wicket_settings' => $bundle['modules']['settings_core'] ?? [],
        'structural_pages' => $bundle['modules']['content_structural'] ?? [],
    ],
];
```

This keeps future engine replacement possible because Portus contract remains independent.
